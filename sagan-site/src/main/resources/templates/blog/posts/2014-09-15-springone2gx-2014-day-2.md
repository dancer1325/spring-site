# SpringOne2GX 2014 - Day 2

_Engineering | Josh Long |  September 15, 2014 | 0 Comments_

![](https://raw.githubusercontent.com/joshlong/meh/master/day2-collage.jpg)

Day 2's kicked off with a bang! My theory is that by that point people had seen a day's worth of sessions, and so the keynote needed to be full of impactful, big-bang announcements and discussions in order to remain memorable. It needed to raise the bar and demonstrate leadership. And it did.

## Spring Framework 4.1

Spring framework lead [**Juergen Hoeller**](http://spring.io/team/jhoeller) kicked things off with an introduction to the new featues in the [recently-released](https://spring.io/blog/2014/09/04/spring-framework-4-1-ga-is-here) [Spring 4.1](http://spring.io/projects/spring-framework).

Spring Framework 4.1 provides annotated JMS listener methods, comprehensive for the final JSR 107 specification, flexible resolution and transformation of static web resources, additional MVC views (like Groovy Server Pages), websocket-support improvements, and many more new features.

## Spring XD

[Spring XD](http://spring.io/projects/spring-xd) co-lead [**Mark Fisher**](http://spring.io/team/mark-fisher) took the stage to introduce the project as it stands now, a year after its debut last year at SpringOne2GX 2013. Mark invited us to consider how far things have come since the early days of Spring, and to consider how interesting the opportunity continues to be today. Then... he told a *joke*: "the past, the present and the future walk into a bar. It was tense."

It took a moment, but the room recovered. Reluctantly. Mark offered to continue as a full time developer and not switch to comedy.

Mark showed us how Spring XD a looks at the future. It is built on top of first-class components like [Spring Batch](http://spring.io/projects/spring-batch), [Spring Integration](http://spring.io/projects/spring-integration), [Spring Data](http://spring.io/projects/spring-data), [Redis](http://redis.io), [RabbitMQ](http://rabbitmq.com), [Zookeeper](http://zookeeper.apache.org), and Netflix's Zookeeper-addition [Curator](http://curator.apache.org/). It offers a simple, linux like domain-specific language (DSL) that requires no compliation, making Hadoop more approachable to beginners and experts alike. Extending Spring XD is done in Spring/Java, often in Spring Integration to adapt to customer's unique systems as Spring XD sources and sinks.

![](https://raw.githubusercontent.com/joshlong/meh/master/spring-xd-unified-platform.png)

It provides powerful data-integration and stream processing capabilities in an operator- and developer-familiar package. Mark introduces a demonstration that analyses the resource consumption of a smart power grid and then, using [PMML](http://en.wikipedia.org/wiki/Predictive_Model_Markup_Language) machine-learning support in Spring XD, predicts future consumption levels. The example uses numerous nodes deployed across multiple cluster nodes, but the assembly and description of the solution in Spring XD itself is really simple! (The crowd seemed to think so, too!)

![](https://raw.githubusercontent.com/joshlong/meh/master/tweet-xd.png)

## Microservices

*\* Dr. Dave Syer*\* (co-founder of [Spring Batch](http://spring.io/projects/spring-batch), co-lead of [Spring Security OAuth](http://spring.io/projects/spring-security-oauth) and [Spring Boot](https://spring.io/projects/spring-boot)) rejoined the stage to follow up on his day 2 update on Spring Boot with some news about, and an introduction to, the new [Spring Cloud](http://spring.io/projects/spring-cloud) umbrella project. Spring Cloud has historically contained Platform-as-a-Service connectors that let you consume services - databases, message queues, etc. - from within a PaaS environment. Spring Cloud now has an expanded scope to define and provide software to better enable modern, *cloud*\-y architectures, like the [microservices](http://martinfowler.com/articles/microservices.html) that Netflix builds atop Spring Boot and [their own, open-source stack](http://netflix.github.io/).

Spring Cloud aims to provide solutions for *emergent patterns* in large scale, often cloud-based applications:

![](https://raw.githubusercontent.com/joshlong/meh/master/dr-syer-microservices-demo-emergent-patterns-of-microservices.png)

The new Spring Cloud project, in fact, integrates popular Netflix OSS components like [Hystrix](https://github.com/Netflix/Hystrix) (which provides implementations of their famous [*reliability patterns*](http://www.slideshare.net/benjchristensen/june-28th-meetup?qid=28efee4d-1ab0-4ed2-9547-86e20eb2f8ea&v=default&b=&from_search=36)) and [Eureka](https://github.com/Netflix/Eureka) (which provides service discovery and location).

![](https://raw.githubusercontent.com/joshlong/meh/master/spring-cloud-components.png)

Dr. Syer introduced a simple microservice that demonstrated two services, one that provided a customers repository and another that provided access to merchants (stores). The customer service returned information about the nearby stores. This information came from the stores service. If the stores service were taken offline, the customers service continued to function.

![](https://raw.githubusercontent.com/joshlong/meh/master/tweet-spring-cloud-netflix.png)

This effort builds upon Spring Boot with an eye towards simplifying the development of microservices. This message seems to have resonated well, and we think that the combination of Spring Boot, Spring Cloud and [CloudFoundry](http://www.cloudfoundry.com) will make a killer combination for JVM apps that need to start life as a monolith, but then need to distrubute and decentralize in order to scale. Adrian Cockcroft explains why distribution and decentralization are so important extremely well in [an interview on InfoQ.](http://www.infoq.com/interviews/adrian-cockcroft-microservices-devops)

![](https://raw.githubusercontent.com/joshlong/meh/master/tweet-netflix-spring.png)

## Reactor

[Reactor Project](https://github.com/reactor) lead [*\*Jon Brisbin \**](http://spring.io/team/jbrisbin) took the stage to then update us on the state of the Reactor project. Reactor has made great strides since we saw it debut last year. It's since become part of a the [Reactive Streams](http://www.reactive-streams.org) effort jointly led by teams at Netflix, Typesafe, Red Hat, Twitter, Oracle, and Pivotal, among others.

It provided the definition for the [`LocationProcessor`](https://github.com/spring-cloud-samples/processor/blob/master/src/main/java/demo/domain/LocationService.java) service in the microservices segment. (The `LocationProcessor` is a nice example of the easy-to-use Java 8 DSL, too!) Reactor is a natural fit for microservices:

![](https://raw.githubusercontent.com/joshlong/meh/master/reactive-architecture.png)

It's a foundational piece or well integrated with Spring Framework 4.0's websocket support, Spring XD, Spring Integration, the upcoming Grails 3.0's event system, and much more. In fact, we saw Reactor in action earlier in the evening's keynote!
